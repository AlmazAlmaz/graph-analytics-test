# Аналитика над графом: PageRank вне оперативной памяти

Репозиторий содержит решение тестового задания на C++20. Я выбрал метрику
PageRank, потому что ее можно считать итерационно, последовательно проходя по
ребрам. Это хорошо ложится на условие задачи: ребра не обязаны помещаться в RAM.

## Структура проекта

```text
src/main.cpp                 реализация PageRank и работы с данными на диске
examples/edges.csv           маленький входной граф для проверки запуска
CMakeLists.txt               сборка проекта через CMake
.github/workflows/ci.yml     сборка и smoke-test на Ubuntu 22.04 в GitHub Actions
.gitignore                   исключения для build-каталогов и временных файлов
```

## Что считается

PageRank оценивает важность вершины в ориентированном графе. Если на вершину
ссылаются важные вершины, ее ранг растет. На практике такая метрика используется
для ранжирования документов, страниц, аккаунтов, объектов в рекомендательных
системах и сущностей в knowledge graph.

Формула одного шага:

```text
next[to] += damping * rank[from] / out_degree[from]
```

Отдельно учитываются dangling-вершины, у которых нет исходящих ребер. Их масса
равномерно распределяется по всем вершинам.

## Почему выбран именно такой способ хранения

Я рассматривал три варианта.

1. Хранить исходный CSV на диске и на каждой итерации перечитывать его. Это
   самый простой вариант, но CSV плохо подходит для многократных проходов:
   парсинг дорогой, а при параллельной записи в `next_rank` нужны атомарные
   операции.

2. Один раз преобразовать CSV в бинарные bucket-файлы. Это выбранный вариант.
   Ребра хранятся на SSD как пары dense-id `(from, to)`. Bucket выбирается по
   вершине `to`, поэтому каждый поток обновляет свой диапазон `next_rank` и не
   конфликтует с другими потоками. В памяти остаются только структуры размера
   `O(V)`.

3. Построить CSR на диске или через mmap. Это быстрее для промышленного решения,
   но заметно сложнее в реализации: нужна внешняя сортировка, аккуратное
   построение offsets и отдельная обработка больших промежуточных файлов.

Я остановился на втором варианте. Он не самый быстрый из возможных, но для этого
задания у него нормальный баланс: ребра не лежат в памяти, итерации работают уже
не с CSV, а с компактным бинарным форматом, и код можно проверить без длинной
подготовки окружения.

## Как устроен алгоритм

Входной файл имеет формат CSV. Используются первые две колонки:

```csv
from,to
1,2
2,3
3,1
4,1
```

Этапы работы:

1. Первый проход по CSV собирает уникальные вершины и исходящие степени.
   Сами ребра при этом не сохраняются в памяти.
2. Вершины сортируются и получают плотные номера `0..V-1`.
3. Строится компактный hash-index `int32 -> uint32`.
4. Второй проход по CSV записывает ребра в бинарные bucket-файлы на SSD.
5. На каждой итерации PageRank потоки параллельно читают bucket-файлы и
   обновляют `next_rank`.
6. После каждой итерации вектор нормализуется, считается L1 delta и проверяется
   условие остановки.

Выходной файл:

```csv
vertex,rank
1,0.332...
2,0.320...
3,0.309...
4,0.0375...
```

## Память и время

Пусть `V` - число вершин, `E` - число ребер, `I` - число итераций.

Память:

```text
O(V)
```

В RAM находятся:

- исходный id вершины и ее `out_degree`;
- плотный индекс вершин;
- два массива PageRank.

Ребра размера `O(E)` лежат на диске в бинарном виде. Один edge занимает 8 байт:
`uint32 from_dense` и `uint32 to_dense`.

Время:

```text
O(E)       первый проход по CSV
O(V log V) сортировка вершин
O(E)       построение bucket-файлов
O(I * E)   итерации PageRank
```

Гипер-узлы не создают пикового расхода памяти. Если у вершины очень большая
исходящая степень, ее ребра просто последовательно читаются и записываются. Если
у вершины очень большая входящая степень, все такие ребра попадают в один
destination bucket, поэтому нет атомарной борьбы за одну ячейку `next_rank`.

## Практические замечания

CSV оставлен входным форматом, потому что он указан в задании и его легко
проверить глазами. Для самих итераций CSV уже не используется: после подготовки
программа читает бинарные bucket-файлы. Это заметно проще, чем каждый раз заново
парсить строки.

CSR я не стал реализовывать сразу. Он быстрее на повторных запусках, но построение
CSR для графа, который не помещается в память, само становится отдельной задачей:
нужна внешняя сортировка ребер и аккуратная работа с временными файлами. Для
тестового проекта это увеличило бы объем кода сильнее, чем улучшило бы саму
демонстрацию подхода.

Временные bucket-файлы не удаляются автоматически. Это сделано намеренно: после
запуска можно посмотреть их размер и убедиться, что ребра действительно вынесены
на диск. Если нужно, каталог можно удалить вручную или передать отдельный
`--work-dir`.

## Ограничения

- В памяти все еще должна помещаться таблица вершин `O(V)`. Если не помещается
  даже она, нужен внешний словарь вершин на диске.
- Bucket-файлы занимают дополнительное место на SSD: примерно `8 * E` байт.
- Дубликаты ребер учитываются как повторные переходы. Если требуется простой
  граф без multiedges, нужна отдельная dedup-стадия.
- CSV-парсер читает первые две колонки. Остальные колонки игнорируются.

## Сборка

Целевая среда из задания:

- Ubuntu 22.04 x64;
- SSD;
- CMake 3.16+;
- компилятор с поддержкой C++20, например `g++ 11+`.

На чистой Ubuntu:

```bash
sudo apt-get update
sudo apt-get install -y build-essential cmake
```

Сборка:

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j
```

## Запуск

```bash
./build/external_pagerank \
  --input examples/edges.csv \
  --output pagerank.csv \
  --iterations 100 \
  --tolerance 1e-12 \
  --threads 4
```

Опции:

```text
--input        путь к CSV from,to
--output       путь к выходному CSV
--iterations   максимум итераций, по умолчанию 20
--damping      damping factor, по умолчанию 0.85
--tolerance    ранняя остановка по L1 delta, по умолчанию 1e-8
--threads      число потоков, по умолчанию hardware_concurrency
--buckets      число bucket-файлов, по умолчанию 4 * threads
--work-dir     каталог для бинарных bucket-файлов, по умолчанию <output>.work
```

## Проверка результата

Для графа из примера:

```bash
./build/external_pagerank --input examples/edges.csv --output pagerank.csv --iterations 100 --tolerance 1e-12 --threads 2
```

Что проверяется:

- программа завершается;
- сумма всех rank близка к `1.0`;
- все rank неотрицательны;
- при `damping < 1` итерации сходятся;
- повторный запуск дает тот же результат с точностью до округления `double`.

Для большого графа команда такая же, меняются только путь к файлу и параметры:

```bash
./build/external_pagerank --input /data/edges.csv --output pagerank.csv --iterations 30 --threads 16 --buckets 128
```
